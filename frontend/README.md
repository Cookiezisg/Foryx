# Anselm Frontend

Flutter desktop client for Anselm. The Go backend remains the source of business
behavior and runs as a localhost sidecar; this package owns the desktop shell,
UI state, generated client-side contract DTOs, and platform launchers.

## How To Read This Directory

The important application code is concentrated in:

| Path | Purpose |
|---|---|
| `lib/main.dart` | Flutter entrypoint. |
| `lib/app/` | Assembly root, app shell, runtime window setup. |
| `lib/core/` | Shared foundation: design tokens, reusable UI, platform helpers, pure models. |
| `lib/i18n/` | Slang source strings and generated typed accessors. |
| `lib/dev/` | Local developer surfaces such as the component gallery. |
| `test/` | Unit and widget tests for the frontend foundation. |
| `assets/` | Bundled brand and font assets. |

The platform folders are Flutter desktop host projects, not product features:

| Path | What It Is |
|---|---|
| `macos/` | Xcode host app, entitlements, app icon, CocoaPods files. |
| `linux/` | Linux GTK/CMake runner generated by Flutter. |
| `windows/` | Windows Win32/CMake runner generated by Flutter. |

Those folders contain Swift, C, C++, CMake, Xcode, and Windows resource files
because Flutter desktop needs a small native host application per operating
system. Most day-to-day frontend work should not touch them unless changing
window chrome, app icon, entitlements, native plugin registration, or platform
packaging.

Generated or local build folders such as `.dart_tool/`, `build/`,
`*/flutter/ephemeral/`, `macos/Pods/`, and `test/dev/out/` are ignored artifacts.

## Commands

Run these from `frontend/`. The toolchain is provided through `mise`.

| Command | Purpose |
|---|---|
| `make setup` | Fetch Flutter/Dart dependencies. |
| `make gen` | Regenerate Slang i18n files. |
| `make analyze` | Run Flutter static analysis. |
| `make test` | Run frontend tests. |
| `make verify` | Run the frontend pre-push gate: gen, analyze, test. |
| `make demo` | Run the desktop app on macOS. |
| `make gallery` | Run the component gallery entrypoint. |
| `make app` | Run the desktop app path intended for sidecar integration. |
| `make shots` | Headless Skia render of the shell + every gallery category → `test/dev/out/*.png` (fast, no Xcode). |
| `make shot-app` | Real-run capture: build + launch the gallery app, screenshot its live window → `test/dev/out/app.png` (Impeller render + OS chrome). |

## Architecture

The frontend follows ADR 0004:

- `app` is the assembly root.
- `core` is shared foundation code.
- future `features/<domain>` folders own their own data, state, and UI.
- Riverpod is the dependency and state container.
- backend contracts are mirrored into Dart DTOs as frontend contract code lands.
- SSE and backend sidecar integration live below feature UI so the UI consumes
  already-shaped streams and repositories.

See `../docs/decisions/0004-frontend-flutter-architecture.md` for the full
architectural decision.
