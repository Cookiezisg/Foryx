## Product-form concerns

- **Scenario fragmentation never landed.** The product story (one knob per stage — chat vs forging vs catalog summary) is great, but the implementation has only `chat` + `web_summary` (the latter falls back to chat anyway). Forge code paths all run via `llmclient.NewClient` → `PickForChat`. Either commit to scenarios and add `forging-*` / `catalog-summary` plumbing, or drop scenario as a concept and just have one config (current de-facto reality). Doc says one thing, code says another.
- **Cost surface is silently large.** 57 tools = ~24k input tokens *every turn*. WebFetch adds another 150k+ tokens per page (invisible to the message-level usage). For a 5-min chat that touches WebFetch + WebSearch + 2 forging tools, real cost is easily 500k+ input tokens *not surfaced to the user*. v1 burn-in didn't catch this because tests used FakeLLM. Real DeepSeek calls reveal it. Add: (a) "verbose token usage" panel that sums all traces in a turn, (b) "tool culling" pass — `Read`/`Write`/`Edit`/`Glob`/`Grep`/`LS` are loaded but chat can't actually use them productively without a workspace concept; consider gating them. (c) per-tool token cost stamp shown in UI next to each tool_call block.
- **Chat does not record reasoning content in history sent to LLM** — I see in `trace #1` (turn 2), the assistant msg from turn 1 has `reasoningLen=0` even though DB stored 50-char reasoning. Some providers (e.g. DeepSeek) want reasoning re-sent so they can build on prior chain-of-thought. This is provider-specific — DeepSeek docs say it's not necessary for chat (only for reasoner). For now, accept. Worth re-testing if behaviour degrades on multi-turn reasoning tasks.
