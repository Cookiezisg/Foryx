## 0. 产品哲学

> **AI 是一个积极参与的协作者，但用户永远能看见 AI 在想什么，并且能纠正它。**

这句话是 Forgify 所有设计决策的判断标准。当出现模棱两可的设计选择时，以此为准。

**第二原则：对话是主角，画布是镜子。**

所有创造和修改都以对话为权威来源。画布是工作流当前状态的实时可视化，不是独立的编辑器。用户可以在画布上直接操作（移动节点、改连线、修改配置），这些操作静默同步到后端，在用户下一次开口时作为整体状态呈现给 AI——AI 读的是当前文档，不是变更日志。

## 1. 产品定位

### 1.1 一句话描述

Forgify 是一个本地运行的桌面应用，让普通人通过对话，把每天重复做的事情锻造成可以自动运行的流程。

### 1.2 名字的含义

**Forge**（锻造）+ **-ify**（...化）——让一切可被锻造。

用户在对话中一点点锻造属于自己的工具和流程。这不是拼装现成零件，而是从原料（需求）开始，在炉火（对话）中反复塑形，最终得到一件趁手的武器。

### 1.3 解决什么问题

现有 AI 工具的三种形态都有局限：

- **对话型 AI（ChatGPT 等）**：给答案，但不能"执行"；用户无法把多次对话的能力沉淀成可复用的流程。
- **可视化编排平台（Dify、n8n）**：功能强大，但门槛高；工具箱固定，无法在对话中即时创造新工具。
- **代码型 Agent（Claude Code）**：面向开发者，终端交互；非技术用户无法使用，且缺乏流程编排能力。

Forgify 填补中间地带：**让非开发者也能通过自然对话，创造属于自己的工具和自动化流程，并对整个过程保持可见和可控。**

### 1.4 目标用户

**主要用户（P0）**：

- **有重复性工作的白领**：运营、行政、财务、采购、市场等岗位，每天处理邮件、Excel、报表、跨系统数据搬运。
- 特征：知道 AI 有用，但不知道如何把多个 AI 能力组合起来；有耐心探索；愿意为自己造工具。

**次要用户（P1）**：

- **想快速自动化的开发者**：懂一点代码，但不想为每个小需求都写完整项目。希望快速把想法落成可运行的流程。
- 特征：能看懂代码，会给 AI 提更精确的要求，是高级工具的创造者和分享者。

### 1.5 核心价值 (First Win)

> **用户看到一个复杂的目标，被 AI 拆解成一个一个清晰的节点，然后通过对话把每个节点攻克，最后串联起来，变成一个真正能跑的流程。**

这个"拆解 → 逐个攻克 → 串联"的过程，就是 Forgify 的魔法时刻。

## 2. 核心概念模型

Forgify 包含三个核心对象：**对话、工具、工作流**。工具和工作流统称为**资产**。

### 2.1 对话 (Chat)

所有创造的起点和入口。

**对话与资产的绑定关系**：

每个对话最多绑定**一个资产**（一个工具或一个工作流）。绑定关系有两种产生方式：

1. **自动绑定**：对话中 AI 创建了一个工具或工作流，该对话自动绑定到这个资产。如果对话中创建了多个资产，以最后一个为准。
2. **固定绑定**：从资产视图内点击"新建对话"，对话从创建起就绑定到该资产。

**绑定的感知与调整**：

- 绑定发生时，对话顶部出现提示条：`📎 此对话已关联到：报价工作流  [切换 ▼] [解除]`
- 用户可点击"切换"重新选择绑定的资产，或点击"解除"变回独立对话
- 绑定解除后，对话仍保留在对话列表，但不再出现在任何资产的关联对话列表中

**对话本身的特征**：

- 每个对话是一个独立的会话，有完整上下文历史
- 对话可以创建、修改工具和工作流（会触发绑定）
- 对话本身是独立的资产，可以被命名、删除
- AI 自动为对话命名（第一轮对话完成后，后台调用轻量模型生成 ≤15 字标题）

**长对话的上下文管理**：

工具锻造和工作流构建通常会很长，Forgify 内置三层上下文压缩：

| 层级 | 触发时机 | 操作 |
|---|---|---|
| **MicroCompact** | 接近上下文限制的 75% | 本地裁剪冗余消息，不调用 API，静默进行 |
| **AutoCompact** | 接近上下文限制的 88% | 调用轻量模型对历史对话做摘要，保留最近 20 条 |
| **FullCompact** | 用户手动触发 | 完整对话总结，历史替换为结构化摘要 |

AutoCompact 发生后，对话顶部显示 `"部分早期消息已压缩" [查看摘要]`。FullCompact 前弹出确认框（不可逆）。

### 2.2 工具 (Tool)

一个封装了具体功能的 Python 函数，可以被工作流调用，也可以在对话中单独测试。

**工具的本质**：每一个工具都是一个可以被对话锻造出来的函数。

**代码规范**：

```python
def send_email(to: str, subject: str, body: str) -> dict:
    """通过 SMTP 发送邮件到指定地址"""
    import smtplib
    # ... 实现
    return {"success": True, "message": "邮件已发送"}
```

规范要求：
- 单一顶层函数，snake_case 命名
- 所有参数必须有类型注解（str / int / float / bool / list / dict）
- 返回值类型为 dict
- 有 docstring 说明功能
- 可使用 import（依赖由 Forgify 自动安装）

**工具的生命周期**：

- `draft`：刚被创建，未测试
- `tested`：最近一次测试通过
- `failed`：最近一次测试失败

**工具的主视图**（在资产页打开）：

- Monaco 代码编辑器，带语法高亮，可直接编辑
- 顶部工具栏：`▶ 测试运行` / `💾 保存` / `导出`
- 右侧 mini-sidebar：与此工具关联的对话列表

**内置工具（第一版）**：

覆盖办公自动化的高频场景：

| 类别 | 工具 |
|---|---|
| 邮件 | Gmail 读/发、Outlook 读/发、邮件附件下载 |
| 文件 | Excel 读/写、Word 读/写、PDF 读取、CSV 处理、本地文件读写 |
| 日历 | Google Calendar 读写、Outlook Calendar 读写 |
| 通讯 | 飞书、钉钉、Slack 消息发送 |
| 网络 | HTTP 请求、网页抓取、Webhook 接收 |
| 数据 | JSON 处理、正则提取、数据转换、排序/过滤 |
| 系统 | 定时触发、文件夹监控、桌面通知 |
| AI | LLM 调用、文本总结、翻译、分类 |

**工具的分享**：

导出为 `.forgify-tool` 文件（JSON 格式），包含函数代码、依赖声明、描述。
导入时显示代码预览和权限说明，用户确认后加入工具库。

### 2.3 工作流 (Workflow)

节点和连线组成的图，每个节点调用一个工具或执行一次 AI 推理。

**工作流的本质**：把多个工具按一定顺序和条件串联起来，变成一个可反复运行的流程。

**节点类型**：

| 类型 | 说明 |
|---|---|
| **触发节点** | 工作流的起点：手动触发 / 定时触发 / 事件触发 / Webhook |
| **工具节点** | 调用一个工具 |
| **Agent 节点** | 给 LLM 一个工具子集 + 目标描述，由 LLM 自主决定调用哪些工具、调用几次、以什么顺序。适合非确定性子任务 |
| **LLM 节点** | 直接调用大模型一次，用于判断、总结、生成文本 |
| **条件节点** | if / else / switch，根据上游数据决定走向 |
| **循环节点** | 遍历列表，对每一项执行内部子流程 |
| **并行节点** | 同时执行多个分支，等待全部完成后汇合 |
| **人工确认节点** | 暂停流程，向 Inbox 发送审批请求，等待用户确认 |
| **子工作流节点** | 调用另一个工作流 |

> **Agent 节点 vs LLM 节点**：LLM 节点是固定的一次调用，给定 prompt 返回文本。Agent 节点是一个自主推理循环，LLM 可以多轮决策，在完成目标前反复调用工具。当子任务的路径不确定时用 Agent 节点，当只需要生成或总结文本时用 LLM 节点。

**工作流的生命周期**：

- `draft`：在对话中正在构建
- `ready`：所有节点配置完整，可运行
- `deployed`：已部署，可被定时或事件触发
- `paused`：已部署但暂停触发
- `archived`：归档，不再使用

**工作流的主视图**（在资产页打开）：

- 画布为主，节点和连线可拖拽
- 顶部工具栏：`运行一次` / `部署配置` / `暂停` / `查看运行历史`
- 节点状态颜色：蓝色（配置完整）/ 黄色（缺失工具）/ 红色（错误）/ 灰色（禁用）
- 右侧 mini-sidebar：与此工作流关联的对话列表

**画布变更的 AI 感知机制**：

- 用户在画布上的操作**静默持久化**，不产生任何对话消息
- 用户下一次发消息时，系统自动将当前工作流状态的紧凑摘要注入 AI（用户不可见）；若画布未改动则跳过注入
- AI 始终基于当前完整状态响应，而不是靠还原变更历史

**工作流运行中的审批（Mailbox 模式）**：

工作流运行时，如果某个节点需要执行级权限（如发送邮件、调用付费 API），不会阻塞整个工作流——而是把这个操作放入 Inbox 的"待审批"队列，工作流中其他无依赖的节点继续并行运行。用户在 Inbox 中批准、拒绝或修改参数后批准。

**部署配置**：

- 触发方式：手动 / 定时（cron 表达式，有可视化编辑器）/ 事件 / Webhook
- 错误处理：失败重试次数、重试间隔、失败后动作（通知 / 停止 / 跳过）
- 通知设置：成功通知、失败通知（路由到 Inbox）

### 2.4 Inbox

Forgify 向用户推送的所有信息的统一入口。

**Inbox 的三类内容**：

| 类型 | 说明 | 可执行的操作 |
|---|---|---|
| **待审批** 🔴 | 工作流运行时遇到需要执行级权限的操作 | 批准 / 拒绝 / 修改参数后批准 |
| **运行结果** | 工作流每次运行的成功或失败通知 | 成功：查看详情；失败：进入错误修复对话 |
| **系统事件** | AutoCompact、工具安装失败等系统级提示 | 查看摘要 / 忽略 |

**Inbox 的优先级**：

- 待审批请求高亮显示，侧边栏 Inbox 图标有红点角标
- 运行失败高亮显示，点击直接进入 AI 已分析完毕的错误修复对话
- 系统事件以普通样式显示，不产生角标

## 3. 界面结构

### 3.1 整体导航

左侧边栏包含四个主入口 + 底部设置：

```
┌───┐
│ 🏠 │  Home
│   │
│ 💬 │  对话
│   │
│ 📦 │  资产（工作流 + 工具）
│   │
│ 📥 │  Inbox  ← 有未读时显示红点
│   │
│   │
│ ⚙️ │  设置（底部固定）
└───┘
```

侧边栏可展开（220px）/ 收起（64px），收起时只显示图标。

### 3.2 Home 页

Home 页是有观点的首页，右侧始终是 AI 对话入口，开口即可开始新对话。

**左侧主内容区**：

**顶部：问候与状态**
- "早上好 / 下午好"（根据时间）
- 一句当天摘要："今天已有 3 个工作流成功运行，1 个失败需要处理"

**第一区块：待处理**
- 从 Inbox 拉取的待审批事项和失败运行（最多显示 5 条）
- 点击直接跳转到 Inbox 或错误修复对话

**第二区块：最近对话**
- 最近打开的 5 条对话
- 每条显示：对话名、绑定的资产名（若有）、最后消息时间

**第三区块：资产概览**
- 最近有活动的工作流（上次运行状态、触发方式）
- 最近修改的工具（状态）

**右侧**：AI 对话输入框，直接开聊即开启新的独立对话。

### 3.3 对话页

**左侧 Panel（对话列表）**：

- 搜索框
- 按时间倒序的所有对话列表
- 每条对话显示：名称、时间、绑定资产的小徽章（如 `📦 报价工作流`）
- 独立对话无徽章

**主内容区（对话视图）**：

顶部（若有绑定）：
```
┌────────────────────────────────────────────────────────────┐
│  📎 已关联到：报价工作流  [切换 ▼] [解除]                  │
└────────────────────────────────────────────────────────────┘
```

中部：消息流，消息类型：
- 用户文本消息
- AI 文本回复（Markdown 渲染）
- AI 操作卡片（"已创建工具 X"、"已更新工作流 Y"）
- 工具测试结果卡片（通过 ✅ / 失败 ❌ + 输出）
- 工作流运行进度卡片（节点执行状态）

底部输入框，支持：
- 文本输入
- 文件拖拽上传（txt / xlsx / pdf / docx / png 等）
- 压缩对话按钮（触发 FullCompact）

**右侧画布**（若绑定了工作流）：
- 显示绑定工作流的实时画布
- 可折叠，折叠后主内容区扩大

## 4. 核心交互流程

### 4.1 首次使用引导

> 待定，回头单独设计。

### 4.2 核心流程：从对话到生产

**场景**：用户想每天自动处理供应商报价邮件。

**步骤 1：表达意图**

用户在 Home 右侧对话框说："我想每天自动处理供应商发来的报价 Excel，汇总后发给采购团队。"

**步骤 2：AI 拆解需求**

AI 回复，提出关键问题：
- 供应商邮件有特征吗（发件人、主题关键词）？
- Excel 格式每家是否一致？
- 汇总的格式是什么？
- 发给谁？自动发还是人工确认？

用户逐一回答。

**步骤 3：工作流生成**

AI 在画布上开始生成工作流节点：

1. `定时触发`（每天 9:00）
2. `读取 Gmail`
3. `下载附件`
4. `[缺失] 解析供应商 A 的 Excel`
5. `[缺失] 解析供应商 B 的 Excel`
6. `[缺失] 解析供应商 C 的 Excel`
7. `汇总数据`（内置工具）
8. `格式化 Excel`（内置工具）
9. `人工确认节点`
10. `发送邮件`

画布实时显示生成过程。对话顶部出现：`📎 此对话已关联到：报价处理工作流`。

**步骤 4：锻造缺失工具**

用户点击节点 4（缺失节点）。AI 说："我需要了解供应商 A 的 Excel 长什么样，能上传一个样例给我吗？"

用户拖入样例文件。AI 分析后生成代码，在对话中以高亮代码块展示，并附"▶ 测试运行"按钮。

用户填写测试参数，运行通过，点击"💾 保存为工具"。工具状态变为 `tested`，节点变为蓝色。

重复完成节点 5、6。

**步骤 5：整体测试**

用户点击工作流顶部的"▶ 运行一次"。

到人工确认节点时，Inbox 收到一条待审批消息，桌面通知弹窗。其他节点继续并行运行（Mailbox 模式）。

用户在 Inbox 查看内容，点击"批准"，工作流从该节点继续。整体运行成功。

**步骤 6：部署**

用户点击"部署"，弹窗确认：
- 触发方式：每天 9:00 ✓
- 权限确认：读取 Gmail、发送邮件 ✓
- 错误处理：失败重试 2 次，失败后发 Inbox 通知 ✓

用户确认，工作流进入 `deployed` 状态。

### 4.3 错误处理流程

**场景**：部署后的工作流运行失败。

**步骤 1：通知**

失败发生，用户在两处收到通知：
- 桌面通知弹窗
- Inbox 出现红色失败条目 + Home 页"待处理"区块

**步骤 2：错误诊断**

用户点击 Inbox 的"进入修复"，直接进入**错误修复对话**（新对话，自动绑定到该工作流）。AI 开场消息已分析完毕：

> "昨晚 9:00 的供应商报价处理失败了。原因是供应商 A 的 Excel 格式变了，多了一列"税率"。
>
> 我有三种处理方式：
> 1. 自动修改解析工具以适应新格式（推荐）
> 2. 你和我一起讨论怎么改
> 3. 先跳过这次，保持工具不变"

**步骤 3：修复**

用户选择方式 1。AI 修改工具代码，运行测试，测试通过，让用户确认。用户确认，工具版本更新，工作流恢复正常。

### 4.4 工具分享流程

**用户 A**：在工具详情页点击"导出" → 选择保存位置 → 生成 `.forgify-tool` 文件 → 分享给用户 B。

**用户 B**：点击资产页"导入工具" → 选择文件 → 弹出预览（工具名、描述、代码、依赖）→ 确认导入，加入工具库（状态为 `draft`，需测试）。

## 5. 权限与安全

### 5.1 权限分级

所有工具在创建时必须声明所需权限，分三级：

**读取级 (Read)**：读取邮件、日历、文件、剪贴板；查询 API、抓取网页。

**修改级 (Modify)**：写入本地文件、修改日历事件；创建邮件草稿（不发送）；修改已有数据。

**执行级 (Execute)**：发送邮件 / 消息；调用付费 API；删除数据；执行外部系统的不可逆操作。

### 5.2 权限确认时机

- **工具创建时**：AI 从代码中分析所需权限，用户首次测试前确认
- **工作流部署时**：部署弹窗中列出所有工具的权限总和，用户确认
- **执行级操作运行时（Mailbox 模式）**：默认路由到 Inbox 等待审批，不阻塞其他节点；用户可设置"信任此工具"后自动批准

### 5.3 敏感数据处理

- **API Key 管理**：AES-256-GCM 加密存储在本地，密钥由机器指纹派生，不离开用户电脑
- **工具分享时**：导出 `.forgify-tool` 时自动剥离 API Key、令牌等敏感数据
- **工具导入时**：显示安全提示，提醒用户导入的代码来自第三方

### 5.4 运行沙箱

- 每个工具在独立的 Python 虚拟环境中运行（uv 管理）
- 默认超时 30 秒，用户可调整（5-300 秒）
- 依赖按工具组合哈希缓存，相同依赖只安装一次

## 6. 模型与 API

### 6.1 模型支持

Forgify 支持所有主流 LLM API：Anthropic Claude、OpenAI / Azure OpenAI、DeepSeek、Moonshot / Kimi、通义千问、任何 OpenAI 兼容 API 端点、Ollama 本地模型。

### 6.2 Key 配置

用户在设置页配置各家的 API Key，可为不同场景指定不同模型：
- **主对话模型**：推理能力强
- **代码生成模型**：代码能力强（工具锻造使用）
- **工作流 LLM 节点默认模型**
- **轻量模型**：用于摘要、自动命名等低成本任务

所有 Key 加密存储，永不外传。

### 6.3 模型调用策略

- 如果当前模型失败（网络、配额、限流），自动降级到备用模型，静默切换 + Inbox 通知
- 代码生成、工作流关键决策默认使用最强推理模型
- 摘要、命名等任务使用轻量模型节省成本

## 7. 本地架构要点

### 7.1 运行方式

- Forgify 是一个桌面应用，基于 **Electron**（Go HTTP 后端 + Chromium 前端）
- 启动后常驻后台，状态栏有图标
- 关闭窗口 ≠ 退出应用，后台继续运行定时任务
- 真正退出需从状态栏菜单选择"退出 Forgify"

### 7.2 数据存储

- 所有数据存储在用户本地（SQLite，WAL 模式，无 CGO）
- 默认存储路径：
  - macOS: `~/Library/Application Support/Forgify/`
  - Windows: `%APPDATA%\Forgify\`
  - Linux: `~/.config/Forgify/`

### 7.3 后台调度

- 内置 Go 调度器负责触发定时任务和事件监听
- 电脑睡眠时任务暂停，唤醒后补跑错过的任务（可配置）
- 多个工作流并发执行，有最大并发数限制

### 7.4 更新机制

- 自动检查更新，用户确认后下载安装

## 10. 成功指标

**激活指标**：
- 用户首次打开后，72 小时内创建出第一个可运行的工作流
- 首个工作流完成后，7 天内至少再创建一个

**留存指标**：
- 部署的工作流持续运行超过 30 天
- 用户每周打开 Forgify 至少 3 次

**深度指标**：
- 用户锻造的自定义工具数量占其使用总工具数的比例
- 工作流的平均节点数与平均运行时长
- Inbox 待审批的平均处理时间

## 11. 未决事项

1. **产品外观语言**：温暖手作风还是工业精密风？"锻造"隐喻在 UI 上如何体现？
2. **工具代码的信任程度**：是否需要对 AI 生成代码做静态分析层（权限声明合规检查）？
3. **错误修复对话的独立程度**：是新开一个对话（带工作流绑定），还是在原对话基础上延续？目前倾向新开。
4. **离线能力**：没有网络时，Forgify 能做什么？应至少能运行已部署的本地工作流。
5. **工作流版本管理**：部署后的工作流改动是否需要版本快照？
