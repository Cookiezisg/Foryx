# G2 · 部署配置 — 产品需求文档

**切片**：G2  
**状态**：待 Review

---

## 1. 背景

工作流从草稿（draft）到实际运行，需要一个明确的"部署"步骤。部署配置让用户确认工作流就绪后才开启自动触发。

---

## 2. 工作流状态机

```
draft → ready → deployed → paused → archived
         ↑                   ↓
         └───────────────────┘
```

| 状态 | 含义 |
|---|---|
| `draft` | 编辑中，不会自动触发 |
| `ready` | 编译通过，可手动运行，未开启自动触发 |
| `deployed` | 自动触发已开启（定时/事件） |
| `paused` | 暂停自动触发，保留配置 |
| `archived` | 归档，不显示在主列表，不可触发 |

---

## 3. 工具栏状态控件

Canvas 顶部工具栏右侧显示当前状态和操作按钮：

```
草稿  [编译检查]  [设为就绪]
就绪  [运行一次]  [部署]
已部署 [运行一次]  [暂停]
已暂停 [运行一次]  [重新部署]
```

---

## 4. 部署前检查

点击"部署"时，系统自动：
1. 运行编译器验证（F1）
2. 检查所有工具节点是否选择了工具
3. 检查 trigger_schedule 节点的 cron 表达式是否合法
4. 检查 API Key 依赖（需要的 key 是否已配置）

任何检查失败时，弹出错误列表，不执行部署。

---

## 5. 部署成功后

- 工作流状态切换为 `deployed`
- 定时触发器立即注册（G1）
- Webhook/文件监听器立即注册（G3）
- Inbox 发送"工作流已部署"通知

---

## 6. 验收测试

```
1. draft 状态 → 工具栏显示"编译检查"和"设为就绪"
2. 点击"设为就绪" → 编译通过后状态变 ready
3. ready 状态 → 点击"部署" → 部署前检查通过 → 状态变 deployed
4. 部署前检查：tool 节点未选工具 → 弹出错误列表，阻止部署
5. deployed → 点击"暂停" → 状态变 paused，定时任务停止
6. paused → 点击"重新部署" → 状态变 deployed，定时任务重新注册
```
