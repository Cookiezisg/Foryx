# H1 · 权限门控 — 产品需求文档

**切片**：H1  
**状态**：待 Review

---

## 1. 背景

工具有不同的权限级别。高风险工具（如发送邮件、写入文件、调用付费 API）在执行前需要确认用户已授权。权限门控是系统级的自动拦截机制，独立于 Approval 节点（设计者主动放置的业务确认点）。

---

## 2. 工具权限级别

| 级别 | 含义 | 默认行为 |
|---|---|---|
| `read` | 只读操作（读文件、查询 API）| 直接执行 |
| `write` | 写操作（写文件、发消息）| 首次执行时确认，记住选择 |
| `execute` | 执行级操作（发邮件、调用付费 API）| 每次执行前确认 |

权限级别在工具的元数据中定义（D6 的 `@permission` 注释字段）。

---

## 3. 确认交互

### 一次性确认（write 级别）

首次执行 `write` 工具时弹出：

```
⚠️ 权限确认
工具 "write_file" 需要写入文件系统的权限。
路径：/Users/xxx/Documents/report.xlsx

[拒绝]  [本次允许]  [始终允许]
```

选择"始终允许"后，该工具不再弹框，直接执行。

### 每次确认（execute 级别）

每次执行 `execute` 工具时都弹出确认（路由到 Inbox，复用 F3 的 ApprovalCard）。

---

## 4. 权限记忆

用户对 `write` 工具的"始终允许"选择存储在本地 DB，下次启动应用后仍然有效。
用户可在设置页面查看和撤销已授权的工具。

---

## 5. 与 Approval 节点的区别

| | H1 权限门控 | F3 Approval 节点 |
|---|---|---|
| 谁放置的 | 系统自动 | 工作流设计者 |
| 触发条件 | 工具权限级别 | 节点存在即触发 |
| 目的 | 安全防护 | 业务流程中的人工判断 |

---

## 6. 验收测试

```
1. read 级别工具 → 直接执行，不弹框
2. write 级别工具首次执行 → 弹出权限确认
3. 选择"始终允许" → 之后直接执行，不再弹框
4. execute 级别工具 → 每次执行前路由到 Inbox 审批
5. 拒绝权限 → 工具不执行，节点返回"权限被拒绝"错误
6. 设置页面可查看和撤销已授权工具
```
