## 1. 这块做什么

用户通过自然语言对话描述需求，AI 在画布上创建和更新工作流节点。对话和画布协同工作，对话是权威来源，画布是实时镜子。

---

## 2. 交互流程

### 2.1 从对话到画布

```
用户说："帮我做一个每天 9 点读取 Gmail、解析 Excel 并发邮件的工作流"
   ↓
AI 提问：确认关键信息（邮件过滤条件、Excel 列名、收件人等）
   ↓
AI 生成 FlowDefinition JSON（工作流的标准格式）
   ↓
Go 层保存到 DB，触发 canvas.updated 事件
   ↓
前端画布重新渲染，节点出现在画布上
   ↓
AI 在对话中说："我已创建了以下工作流结构：..."
```

### 2.2 画布状态注入（整体同步）

每次用户发消息时，如果画布自上次 AI 回复后被修改（用户或 AI 都算），系统自动在用户消息前插入隐藏上下文：

```
[当前工作流状态，14:32 更新]
节点：定时触发(9:00) → 读取Gmail → 解析Excel → 人工确认 → 发送邮件
状态：draft，5个节点，4条连线
```

### 2.3 用户主动修改画布

- 拖拽节点位置 → 静默保存
- 删除节点 → 静默保存，画布更新
- 修改节点配置 → 静默保存
- 以上操作**不产生对话消息**
- 下一次用户发消息时，AI 通过隐藏上下文感知到变化

---

## 3. FlowDefinition 格式

工作流的内部 JSON 格式，AI 操作的数据结构：

```json
{
  "id": "workflow_abc",
  "name": "供应商报价处理",
  "nodes": [
    {
      "id": "node_1",
      "type": "trigger_schedule",
      "position": {"x": 100, "y": 100},
      "config": {"cron": "0 9 * * *"}
    },
    {
      "id": "node_2",
      "type": "tool",
      "position": {"x": 300, "y": 100},
      "config": {
        "tool_id": "gmail_read",
        "params": {"query": "from:supplier subject:报价"}
      }
    }
  ],
  "edges": [
    {"id": "edge_1", "source": "node_1", "target": "node_2"}
  ]
}
```

---

## 4. 对话绑定

工作流创建后，当前对话自动绑定到该工作流（触发 C1 的绑定逻辑），对话顶部出现绑定指示条，画布出现在右半区。

---

## 5. 验收标准

- [ ] 对话中描述需求 → AI 创建节点 → 画布实时出现
- [ ] 对话移动节点位置 → 刷新后位置保留
- [ ] 下一轮对话"再加一个条件节点" → AI 只更新对应部分
- [ ] 对话自动绑定到创建的工作流
- [ ] 用户修改画布后发消息 → AI 的回复中体现了对当前状态的感知
