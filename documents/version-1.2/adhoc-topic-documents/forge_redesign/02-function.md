## 1. 定位

Function 是 Trinity 中**最轻**的产物 —— stateless 一次性 Python 函数,**跑完即销**。

| 属性 | 值 |
|---|---|
| 状态 | 无(每次跑新 subprocess) |
| 调用频次 | 一次性 / 反复都 OK |
| Lifetime | 单次调用(execution-scoped) |
| 形态 | LLM 写的 Python 函数 |
| 承载 | sandbox v2 Python EnvManager + per-call temp owner |

### 与现有 forge 的关系
**完全替代,推倒重做**(决策见 [`00-overview.md`](./00-overview.md) §2.1)。新 entity / table / API / LLM tool 全套独立设计。无数据迁移路径(forge 是开发期数据,V1 上线前清空)。

## 2. 锻造 / 调用模型

### 2.1 锻造 — `create_function` / `edit_function` ops 流式 + 内部 env-fix loop

LLM 通过 ops 流式构建 Definition。每个 op 应用 emit 一个 progress delta(详见 [`01-shared-tool-interface.md`](./01-shared-tool-interface.md) §4.2)。

Definition 包含:
- `name` / `description` / `tags`
- `code`(Python 函数主体)
- `parameters`(JSON schema,入参约束)
- `returnSchema`(JSON schema,返回约束)
- `dependencies`(PEP 508 specifier list)
- `pythonVersion`(PEP 440 spec,可空走默认 `>=3.12`)

**环境同步发生在 tool 内部**(D-redo-9 / D-redo-15):`create_function` / `edit_function` 单次工具调用就闭环:apply ops → 装 env → 若失败调主 chat scenario LLM(DeepSeek)让它建议新 deps → 销旧 env 重装 → 直至成功 or maxAttempts(默认 3)。主 chat 只看最终 tool_result,中间 retry 是 tool 自治。详 [`discussions/2026-05-12-env-and-sse-rework.md`](./discussions/2026-05-12-env-and-sse-rework.md) §E。

**Create 首次 + 校验通过 + env=ready → auto-accept**(对齐 forge TE-15)。env 装失败时返 envStatus=failed + envError + attemptsUsed,**不 auto-accept** — Definition 行已建但 active_version_id 指向 failed 版本,LLM 看到 tool_result 后决定下一步(通常 retry edit_function 或跟用户解释)。

**Edit — "iterate same pending"**(D-redo-11):
- 无 pending → 在 active 之上 ApplyOps → **创建新 pending Version 行** → 装 env(走 fix loop)
- 有 pending → 在 pending 之上 ApplyOps → **重写同 ID pending 行**(不创建新行)→ 销旧 env → 装新 env(走 fix loop)
- 这让 LLM 反复 patch 同一份 draft 直到 env ready,而不是"每次 edit 创建新行"造成版本表泛滥
- **`ops=[]` 显式语义 = 强制重建 env**(D-redo-22):不改任何字段,直接销当前 active version 的 env + 重装(走 fix loop)。用于 "env 突然坏了想 fix" 路径,取代以前用 fake set_meta 触发重装的 hack

**Accept / Reject / Revert**:
- `AcceptPending` 是**纯指针翻转**(D-redo-10):env 已在 edit 阶段装好,accept 不再 sync,瞬时返
- `RejectPending` 调 sandbox.DestroyEnv + 删 pending Version 行(D-redo-12)
- `Revert` 翻 active_version_id 指针到目标 accepted version;目标 env 被 sandbox GC evict 时,run 路径 in-flight 重建(D-redo-13)

### 2.2 调用 — `run_function`

每次调用:
1. Resolve active version
2. **env 已 ready**(锻造阶段已装好)→ spawn 直接进
3. **env=evicted**(罕见,sandbox GC 触发)→ in-flight 同步重建后再 spawn
4. Spawn Python subprocess(driver 模板 + user code)
5. 通过 stdin 喂 args JSON
6. stdout 读 result JSON
7. **Subprocess 跑完即销**

**每版本独立 venv**(D-redo-8):每个 Version 行**独立生成**自己的 env_id(`fnenv_<16hex>`),跟 version_id 是 1:1 但**不等同** — env_id 是 function 自己的命名空间,sandbox 当不透明 string。**不再按 (deps, python) 哈希共享 venv**。uv global wheel cache 让磁盘开销不放大太多(同包跨版本共享 wheel 二进制层),换取 envStatus 字段对当前 version 状态零歧义 + 销 env 不影响其他版本 + sandbox 跨域复用时 EnvID 命名互不干扰(handler 用 `hdenv_`、mcp / chat tool calls 用各自前缀)。

## 4. Python 代码契约

锻造时 LLM 写出的 Python 函数必须满足:

1. **函数名跟 Function `name` 一致**(driver 自动 import 调用)
2. **单个公开函数**(其他辅助函数允许私有 `_`-prefixed)
3. **入参跟 `parameters` spec 对齐**(driver 按 schema 装载 args 调用)
4. **返回 JSON-serializable**(driver 序列化输出)
5. **不能 import Handler client**(决策 D7)— 校验时静态扫 import 黑名单
6. **PathGuard 守敏感路径**(沙箱 v2 现有约束)

### 4.1 driver 模板

```python
# Auto-generated by Forgify; do not edit.
import sys, json, traceback
sys.path.insert(0, '/sandbox/lib')

from user_function import {function_name}  # noqa: E402

def main():
    args = json.loads(sys.stdin.read())
    try:
        result = {function_name}(**args)
        sys.stdout.write(json.dumps({"ok": True, "output": result}))
    except Exception as e:
        sys.stdout.write(json.dumps({
            "ok": False,
            "error": str(e),
            "trace": traceback.format_exc()
        }))

if __name__ == "__main__":
    main()
```

### 4.2 用户函数示例

```python
# user_function.py — 由 LLM 锻造
def markdown_to_pdf(text: str, title: str = "Document") -> dict:
    """Convert markdown to PDF, return {pdf_path, page_count}."""
    import markdown, weasyprint
    html = markdown.markdown(text)
    full_html = f"<h1>{title}</h1>{html}"
    pdf_bytes = weasyprint.HTML(string=full_html).write_pdf()
    
    import tempfile, os
    fd, path = tempfile.mkstemp(suffix=".pdf")
    os.write(fd, pdf_bytes)
    os.close(fd)
    
    return {"pdf_path": path, "page_count": 1, "size_bytes": len(pdf_bytes)}
```

## 5. 持久化 — 2 张本域表 + 1 张 execution log 表

本域**自身**只 2 表(functions / function_versions,以下 §5.1 / §5.2);执行记录(`function_executions` 表)走**共享 schema 模板**,详见 [`08-executions.md`](./08-executions.md) §4.1。Service.Run 终态写入。LLM 通过 `query_executions({kind:"function", entityId:fn_xxx})` 看历史。

### 5.1 `functions`

| 字段 | 类型 | 说明 |
|---|---|---|
| id | TEXT PK | `fn_<16hex>` |
| user_id | TEXT 索引 | local-user |
| name | TEXT | partial UNIQUE `(user_id, name) WHERE deleted_at IS NULL` |
| description | TEXT | 一句话说明 |
| tags | TEXT(JSON 数组) | 可选 |
| active_version_id | TEXT | 指向当前活 FunctionVersion(草稿期空) |
| created_at / updated_at / deleted_at | — | GORM 标配 |

**计算字段**(`gorm:"-"`):
- `Pending *FunctionVersion`(由 service `attachPending` 在 GET / List 后填)
- `EnvStatus / EnvError / EnvSyncedAt / EnvSyncStage / EnvSyncDetail`(从 active version 拷)

### 5.2 `function_versions`

| 字段 | 类型 | 说明 |
|---|---|---|
| id | TEXT PK | `fnv_<16hex>` |
| function_id | TEXT 索引 | FK → functions.id |
| status | TEXT CHECK | `pending` / `accepted` / `rejected` |
| version | INT NULL | accepted 时递增,pending/rejected 为 NULL |
| code | TEXT | Python 函数主体 |
| parameters | TEXT (JSON) | List[ParameterSpec] |
| return_schema | TEXT (JSON) | JSON Schema object |
| dependencies | TEXT (JSON) | List[string] PEP 508 |
| python_version | TEXT | PEP 440(空回退 `>=3.12`) |
| env_id | TEXT 索引 | `fnenv_<16hex>`,每 Version 行独立生成(D-redo-8 与 version_id 解耦) |
| env_status | TEXT | pending / syncing / ready / failed / evicted(白名单 service 层校验) |
| env_error | TEXT | uv stderr / env-fix loop 末态 error 摘要 |
| env_synced_at | DATETIME | success 时戳 |
| env_sync_stage | TEXT | resolving / preparing / installing / fixing(sync 期) |
| env_sync_detail | TEXT | uv stderr 当前 stage 行 / fix loop attempt 信息 |
| change_reason | TEXT | "edit_function: ..." / "manual edit" / "reverted to v{N}" / "initial" |
| created_at / updated_at | — | GORM 标配 |

accepted 版本上限 50 条/function,超限硬删最旧(对齐 forge_versions)。

**每 Version 独立 env 的代价 + 收益**(D-redo-8):
- 代价:venv 目录数 ≈ 总 version 数(每 function 上限 50 + 各家 pending,小尺度);
- 收益:`env_status` 字段对当前 version 状态零歧义 — 旧设计下多 version 共享 EnvID,一个 version 装包成功 ≠ 另一行的 env_status 翻 ready,UI 在撒谎;
- 命名空间隔离:EnvID 由 function 自己生成(`fnenv_`),sandbox 当不透明 string;handler 用 `hdenv_`、mcp / chat tool 等其他消费者用各自前缀互不干扰 — sandbox 是共享基础设施,不绑定具体消费者的 entity ID 语义;
- 磁盘:uv global wheel cache 让同包跨版本共享 wheel 二进制层,实际增量 ≈ 单 venv 的元数据(Python 已 install 的 metadata + 软链),不是完整重装。

## 6. Function 不能调 Handler — D7 决策

### 决策
**Function 内 Python 代码不允许 import / 调用 Handler client**。

### 理由
- Function 是纯叶子节点(stateless),组合靠 workflow 节点编排
- 防递归 / 防循环依赖
- 强制可视编排(用户能从 workflow 图上看到 function ↔ handler 的调用关系,不藏在 Python 里)
- 跟设计哲学一致(D5 各 domain 隔离)

### 实施
- **静态扫 import 黑名单**:`set_code` op 应用时扫 import 列表,出现 `from forgify_handler import` 类的语句直接 reject
- **运行时拦截**:driver 运行时不注入任何 Handler client(没法 import 起)

### 用户场景如何替代
- "Function 想用数据库" → 不写 SQL 函数,而是 workflow 节点 `handler.pg.query` + `function.process` 串起来
- "Function 想发 HTTP" → workflow 节点 `http` + `function.parse` 串起来,或 function 内自己用 Python `requests`(不是 Handler,是 stdlib / 第三方库 — 允许)

## 7. LLM 工具集

```typescript
search_function({ query?, limit?, cursor? })
  → { items: FunctionSummary[], nextCursor?, hasMore }

get_function({ id })
  → { function: Function, activeVersion?: FunctionVersion, pending?: FunctionVersion }

create_function({ name, description, ops, changeReason })
  → { id, versionId, version, status, envStatus, envError?, attemptsUsed, opsApplied }
  // 流式:每 op emit progress delta;env 装阶段 emit forge_env_attempt × N;
  // 内部 env-fix loop:装失败 → 调主 chat scenario LLM 让其只改 deps → 重装 →
  // 最多 3 次(D-redo-15/16/17/18);成功后 auto-accept v1。

edit_function({ id, ops, changeReason })
  → { pendingId, envStatus, envError?, attemptsUsed, opsApplied }
  // 流式:同 create。
  // pending 已存在时 — 重写同 ID pending 行(D-redo-11 iterate same pending),
  // 不返 ErrPendingConflict;旧 env 销 + 新 env 装走 fix loop。
  // ops=[] 显式语义 — 强制重建当前 active version 的 env(D-redo-22),
  // 不改任何字段,直接重装走 fix loop;给"env 突然坏了"路径用。

revert_function({ id, targetVersion })
  → { activeVersionId }
  // 翻 active 指针;目标 version 的 env 已 evict 时 in-flight 重建

delete_function({ id })
  → { deleted }

run_function({ functionId, args, version? })
  → { ok, output, error?, elapsedMs }
  // 锻造阶段已装好 env,run 不再触发装包;
  // env=evicted 时 in-flight 重建后再 spawn;
  // 流式:Python stdout/stderr 行 emit progress

// D22 — execution log 工具(per-entity)
search_function_executions({ functionId?, versionId?, status?, conversationId?, flowrunId?, since?, until?, limit?, cursor? })
  → { count, executions[], nextCursor?, aggregates }
get_function_execution({ id })
  → { ...全字段..., input 截 4KB, output 截 4KB, hints }
```

详见 [`01-shared-tool-interface.md`](./01-shared-tool-interface.md) §2 / §4 + [`08-executions.md`](./08-executions.md) §7 + [`discussions/2026-05-12-env-and-sse-rework.md`](./discussions/2026-05-12-env-and-sse-rework.md) §E。

## 8. HTTP API(~12 端点)

```
POST   /api/v1/functions                          创建(直接传 Definition,LLM 工具走 ops)
GET    /api/v1/functions                          列表
GET    /api/v1/functions/{id}                     详情(含 pending)
PATCH  /api/v1/functions/{id}                     改 name / description / tags
DELETE /api/v1/functions/{id}                     软删

POST   /api/v1/functions/{id}:run                 执行(取消走 caller ctx,无 per-call timeout)

GET    /api/v1/functions/{id}/versions            版本列表
GET    /api/v1/functions/{id}/versions/{v}        单版本详情
POST   /api/v1/functions/{id}:revert              回滚

GET    /api/v1/functions/{id}/pending             看 pending
POST   /api/v1/functions/{id}/pending:accept      接受(瞬时返,env 不在此装)
POST   /api/v1/functions/{id}/pending:reject      拒绝(销 env + 删 pending 行)

GET    /api/v1/functions/{id}/executions          执行历史(D22;详 [`08-executions.md`](./08-executions.md) §6.1)
```

**删除的端点**:`:resync` 端点删除(D-redo-14)。"env 装失败后想重装"路径走 LLM tool `edit_function({id, ops:[]})`(D-redo-22),不再有 HTTP 直接入口。HTTP `:run` 不再有 `timeoutMs` 参数 — 取消由 caller HTTP 断连推动。

## 9. 错误码

| Code | HTTP | Sentinel | 触发 |
|---|---|---|---|
| `FUNCTION_NOT_FOUND` | 404 | `function.ErrNotFound` | id 查不到 |
| `FUNCTION_NAME_DUPLICATE` | 409 | `function.ErrDuplicateName` | 重名 |
| `FUNCTION_VERSION_NOT_FOUND` | 404 | `function.ErrVersionNotFound` | revert/get version 版本不在 |
| `FUNCTION_PENDING_NOT_FOUND` | 404 | `function.ErrPendingNotFound` | accept/reject 无 pending |
| `FUNCTION_RUN_FAILED` | 422 | `function.ErrRunFailed` | sandbox 内部错(≠ ok=false) |
| `FUNCTION_AST_PARSE_FAILED` | 422 | `function.ErrASTParseError` | 代码 AST 不通过 |
| `FUNCTION_NO_ACTIVE_VERSION` | 422 | `function.ErrNoActiveVersion` | 无 active(草稿 + pending 未 accept) |
| `FUNCTION_ENV_NOT_READY` | 422 | `function.ErrEnvNotReady` | env status=syncing/evicted/pending |
| `FUNCTION_ENV_FAILED` | 422 | `function.ErrEnvFailed` | env-fix loop 跑满 maxAttempts 仍失败;envError 含末态摘要 + attemptHistory |

## 10. Sandbox 集成

每个 FunctionVersion 绑一个独立 venv(D-redo-8):
- `env_id = idgenpkg.New("fnenv")`(每 Version 独立 + 跟 versionID 1:1 但解耦),跨 version 不共享 venv;销 env 不影响其他版本
- uv global wheel cache 让同包跨 version 共享 wheel 二进制层(磁盘开销主要在元数据)
- EnvManager=python(sandbox v2 现有)
- Spawn 时调 `sandboxapp.Service.Spawn(owner, ...)`,owner = `function-call:<callId>`

**Env 装配时机** — **同步发生在 `create_function` / `edit_function` 工具内部**(D-redo-9):
1. apply ops → 持久化 Version 行(env_status=`syncing`)
2. 调 sandbox sync(blocking)→ 若成功 → `ready`
3. 失败 → 进 env-fix loop(详 §2.1 + 讨论文档 §E)
4. loop 跑满或某轮 LLM 调用失败 → `failed` + envError + attemptsUsed(D-redo-21)

**Service.Create / Edit 调 sandbox 前 ping**(D-redo-20):ping 失败 → 立即返 `ErrSandboxUnavailable`(503),**不建 Definition 行 / 不建 Version 行**。Sandbox 是项目基石,bootstrap 失败 = 全局瘫痪,不做"graceful 占位"。

**没有后台 sync goroutine**(D-redo-14):删 `SyncEnvForVersion` fire-and-forget 路径 + `Resync` 异步入口。所有 env 状态变更要么在 tool 调用线程内同步发生,要么在 run/spawn 时 in-flight 重建(evict 路径)。

## 11. Catalog source

```go
// app/function/catalog_source.go
func (s *Service) AsCatalogSource() catalogdomain.CatalogSource {
    return &catalogSource{svc: s}
}

func (cs *catalogSource) Name() string { return "function" }
func (cs *catalogSource) Granularity() catalogdomain.Granularity { return catalogdomain.PerItem }
func (cs *catalogSource) ListItems(ctx context.Context) ([]catalogdomain.Item, error) {
    fns, _ := cs.svc.List(ctx, ...) // active + non-deleted
    items := make([]catalogdomain.Item, 0, len(fns))
    for _, fn := range fns {
        items = append(items, catalogdomain.Item{
            Source:      "function",
            Name:        fn.Name,
            Description: fn.Description,
            Category:    "computation", // optional
        })
    }
    return items, nil
}
```

main.go 装配:`catalogService.RegisterSource(functionService.AsCatalogSource())`。
