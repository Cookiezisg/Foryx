## R1 [HIGH] (subprocess-reaping) — ✅ FIXED

**Background bash processes (and their whole process trees) are never reaped on shutdown — Manager.Stop() is unreachable**

Over a long single-user session this is also reachable each time the resident instance crashes and is respawned via the Boot/Get path. Accumulation: each wedged spawn leaks one orphan python subprocess + one blocked goroutine that never die (directly matching the seed defect's '~10 leaked python handler drivers' — a hung Init both fails to register the instance for later Shutdown AND leaves the process running).

## R5 [HIGH] (channel-deadlock) — _pending_

**Durable SSE Publish on a never-cancelling Detached context blocks forever behind one wedged (connected-but-not-reading) client, freezing the whole workspace stream bus**

- **位置**: `backend/internal/infra/stream/bus.go:106-113 (durable fan-out); trigger paths: backend/internal/app/chat/emit.go:91-107 + backend/internal/app/chat/host.go:141-158; backend/internal/app/notification/notification.go:86-90; backend/internal/pkg/reqctx/workspace.go:61-63`
- **反模式**: unbuffered/bounded send under a lock with no escape when the publisher ctx never cancels (send to a slow/stuck-but-present receiver)
- **触发/累积**: A single SSE client that is connected but not draining: a frozen/backgrounded Flutter desktop window, a suspended laptop, a paused client-side debugger, or just a client on a slow/stalled link. The frontend keeps all three SSE streams open for the entire session and they are 'never idle' (build.go:182 comment), so the wedged-subscriber window is long-lived. Any assistant turn finishing (WriteFinalize → durable message_stop on Detached ctx) or any notification/entity durable frame then blocks under st.mu.
- **证据**: Bus.Publish's DURABLE path fans out under st.mu (the per-workspace lock) with exactly three escapes:

      select {
      case s.ch <- env:
      case <-s.done:        // only closed when the subscriber's HTTP ctx is Done
      case <-ctx.Done():    // the PUBLISHER's ctx
      }
  }
Open/Close/non-ephemeral Signal are durable (frame.go:85-88). The subscriber channel cap is sseBufSize(256)+subscriberHeadroom(256)=512 (subscribe.go:32, build_data.go:89). At that point a durable Publish can only proceed via `<-s.done` (client fully disconnects → r.Context() Done → subscribe.go:74-78 cancel → close(done)) or `<-ctx.Done()` (publisher ctx). The lethal combination: MANY durable publishes run on reqctxpkg.Detached() contexts, which are built from context.Background() and DELIBERATELY never cancel (workspace.go:57-62 comment: 'from context.Background() (NOT WithoutCancel) … so the parent request's cancel cannot interrupt it'). With a wedged client, this send has no live escape: subscriber ch is full, s.done not closed (still connected), ctx.Done never fires (Detached). It blocks holding st.mu, so EVERY other Publish to that workspace (chat deltas via loop/emit.go, entity build frames, all notification pushes) serializes behind st.mu and also blocks. notification push (notification.go:86) is likewise a durable Signal and is emitted from background/detached paths.
- **修法**: Durable fan-out to a per-subscriber channel must not be able to block the shared producer indefinitely. (3) Never publish durable frames on a context that cannot cancel: thread a bounded-timeout context (context.WithTimeout off Detached) into WriteFinalize/notification.push so ctx.Done() is a guaranteed escape. Option (1)+(2) together is the robust fix: bound per-subscriber send time and don't hold the workspace lock across a network-paced send.

## R8 [MEDIUM] (unbounded-memory) — _pending_

**Webhook trigger routes are never unmounted from the stdlib http.ServeMux — every distinct path ever registered leaks a handler closure**

- **位置**: `internal/infra/trigger/webhook/webhook.go:106-114 (Register mounts via mux.HandleFunc), 122-129 (Unregister deletes registry/paths but NOT the mux route); comment at 107 admits 'stdlib mux can't unmount'`
- **反模式**: append-only route registration on a long-lived http.ServeMux (no unmount API)
- **触发/累积**: A user editing a webhook trigger's `path` (each edit mounts the new path, old never freed) or delete+recreate of webhook triggers — every new triggerID yields a unique `/api/v1/webhooks/<triggerID>/<path>` so each is a fresh permanent mux entry. ACCUMULATES: yes, monotonic for the process lifetime. Each entry is small (a closure capturing the path string), so it is slow RAM creep; the latent panic on path reuse after Unregister is the sharper risk.
- **证据**: Register calls `l.mux.HandleFunc(full, l.handleWebhook(full))` (line 112) the first time a `full` path is seen. Unregister and the path-change branch only `delete(l.registry, ...)` / `delete(l.paths, ...)` — the mux route stays mounted forever (the code even documents 'stdlib mux can't unmount; leftover route 404s via registry miss'). So the ServeMux's internal entry map accumulates one permanent entry per distinct webhook path ever registered. Worse, because the `!mounted` guard (line 111) keys on `l.registry` (which IS cleared on Unregister), re-registering a path that was previously unregistered re-enters HandleFunc → stdlib `ServeMux.Handle` PANICS on a duplicate pattern ('multiple registrations for ...').
