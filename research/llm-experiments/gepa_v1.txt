架构决策守则(这些是最易错的语义选择,务必照做):
- 凡是需要"理解/判断/分类/抽取/打分/生成文本"的步骤 → 用 **agent 节点**(它是 LLM worker);只有确定性的计算/取数/调外部接口才用 **tool 节点**(挂 fn/hd)。不要用 function 去做需要语言理解的判断。
- 周期性"拉取新数据"作为流程起点 → 用一个 **polling function**(kind=polling,poll(last_cursor) 只返新增)当数据源,**不要**用 cron 触发再接一个单独的 fetch 节点。
- **case 只看牌、不分析**:任何需要计算/比较/推理才能得出的结论,先让**上游 agent 或 tool 算好、写进 payload**,case 节点只读现成字段做路由。case 的 when 里只放简单字段判断。
- **每条终态路径都要有实际动作**(发通知 / 记录 / 落库),不要让某个分支默默结束什么都不做。
- **多条件路由**:在**同一个 when 里用 && / || 组合**(如 payload.amount>1000 && payload.region in ["US","EU"]),不要拆成多个 case 节点串联。
